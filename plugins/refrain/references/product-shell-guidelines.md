# Product Shell Guidelines

Refrain product shells should feel like tools people can live in.

## Layout Defaults

- Prefer a quiet sidebar plus a primary work area.
- Keep the first screen operational, not promotional.
- Use tabs for peer views such as Preview, Code, Files, History, and Quality.
- Keep settings split by function: connection, model, appearance, storage.
- Expose run state close to the action that started it.

## Density

Use compact controls and readable spacing. A tool can be calm without being
empty. Avoid large decorative hero sections unless the product is explicitly a
landing page.

## Components

- Icon buttons need accessible labels and tooltips when meaning is not obvious.
- Segmented controls are for modes.
- Select menus are for option sets.
- Sliders, steppers, and numeric inputs are for quantities.
- Cards are records, dialogs, or framed tools; do not put cards inside cards.

## States

Every data surface needs:

- loading
- empty
- error
- populated
- stale or disconnected when runtime data can drift

Empty states should offer a next action, not explain the product.

## Color and Motion

- One accent.
- Neutral surfaces.
- Borders first.
- Shadow only for overlays.
- Motion under 180ms by default.
- No bounce, elastic, or decorative looping motion.

## Copy

Use concrete nouns and verbs:

- "Scan agents"
- "Build"
- "Preview"
- "Restore version"
- "Open log"

Avoid:

- "Experience the future"
- "Unlock your workflow"
- "Supercharge creativity"
- "Seamless collaboration"
