# Settings

Settings are operational controls, not a dumping ground.

## Structure

- sidebar by concern
- provider or runtime identity
- connection/status check
- defaults
- local paths or storage
- appearance only when it affects product use
- destructive actions isolated

## Rules

- Surface status close to the setting that controls it.
- Keep secret fields and local paths readable but protected.
- Use test/check actions for external dependencies.
- Keep explanatory copy short and specific.

## Reject

- single long settings page
- destructive controls near setup controls
- abstract preference names
- hiding runtime failure behind a generic failed badge
